# FAQ

Every answer below is grounded in code in this repository. Where the honest answer is "not yet implemented," that is stated explicitly and marked **PLANNED**.

---

**Q1. What does this project actually do?**

It runs an HTTP gateway that Claude Code talks to using the Anthropic Messages API, and translates each request into an OpenAI-compatible chat-completions request for whichever upstream provider your `config.json` routes it to (`internal/translate/anthropic.go:1-6`). It is a clean-room Go reimplementation of the Node `@musistudio/claude-code-router` (`NOTICE:1-11`).

**Q2. Why does HTTP/3 require TLS?**

QUIC — the transport HTTP/3 runs on — has no cleartext mode; encryption is baked into the protocol itself, not layered on top the way TLS sits on top of TCP for HTTP/1.1/2. The gateway enforces this at startup: if `EnableHTTP3` is set without both `CertFile` and `KeyFile`, `Start()` returns an explicit error rather than silently falling back to plaintext HTTP/1.1 while still claiming HTTP/3 support — that would misreport the transport actually in use (`internal/gateway/gateway.go:223`, tested at `internal/gateway/gateway_test.go:165-174`).

**Q3. Why is brotli preferred over gzip?**

Because it compresses JSON and Server-Sent Events markedly better, and Claude Code's traffic is almost entirely one or the other. `negotiate()` returns `"br"` whenever the client's `Accept-Encoding` lists it as acceptable at all — it does not compare `q=` weights between brotli and gzip, only whether each is acceptable (`q=0` excludes it); brotli simply wins by capability, not preference score (`internal/gateway/compress.go:39-81`, tested explicitly with `"br;q=0.1, gzip;q=0.9"` still resolving to `br` at `internal/gateway/gateway_test.go:35`).

**Q4. Why are `Providers` and `Router` capitalised in `config.json`, when that's not idiomatic Go/JSON?**

Because that is the wire format the Node implementation, and every existing `claude_toolkit`-managed installation, already use. Renaming the keys to lowercase would be more idiomatic but would silently break every installed config on the day of upgrade — so the capitalisation is preserved exactly, on purpose (`internal/config/config.go:16-18`).

**Q5. What do the `cleancache` and `streamoptions` transformers do?**

- `cleancache` strips Anthropic-only `cache_control` blocks from the outgoing request. Upstreams that don't recognise the field tend to reject the **entire** request rather than ignore the unknown key, so every occurrence — at any nesting depth (system blocks, message content blocks, tool definitions) — has to go. `translate.StripCacheControl` walks the generic JSON tree recursively for exactly this reason (`internal/translate/anthropic.go:538-585`, tested at `internal/translate/anthropic_test.go:194-220`). It decodes with `json.Decoder.UseNumber()` rather than a plain `json.Unmarshal` into `any`, specifically so it never touches any number literal it passes through: a plain unmarshal converts every JSON number to `float64`, which both rejects any literal whose magnitude overflows `float64` (a fuzz-discovered case, `"1E700"`) and — worse — silently corrupts a large integer id like a 20-digit snowflake id into lossy scientific notation.
- `streamoptions` adds `stream_options.include_usage: true` to the OpenAI request, but **only** when the request is actually streaming — some upstreams 400 on a non-streaming request that carries `stream_options` at all. Without it, a streamed response would report no token usage whatsoever on completion (`internal/translate/anthropic.go:351-353`, tested at `internal/translate/anthropic_test.go:173-190`).

Both are opt-in per provider via `"transformer": {"use": [...]}`  (`internal/config/config.go:66-72`), and both are **fully wired end-to-end today**. The live `POST /v1/messages` handler reads `provider.Has("streamoptions")`/`provider.Has("cleancache")` directly into `Options.StreamOptions`/`Options.CleanCache` (`internal/gateway/messages.go:249-250`), and `AnthropicToOpenAI` acts on both.

For `cleancache` specifically: the typed Anthropic→OpenAI struct conversion already drops `cache_control` naturally for every content path it models (`OpenAIMessage`/`OpenAITool` simply have no such field) — that part needed no flag at all. What the flag actually gates is the one place the typed conversion *cannot* reach: a tool's `input_schema`, which travels through as an untouched `json.RawMessage` and can itself carry a `cache_control` key. `AnthropicToOpenAI`'s tool-conversion loop now calls `translate.StripCacheControl` on `input_schema` whenever `opt.CleanCache` is set (`internal/translate/anthropic.go:441-461`) — schema-aware, so a tool that legitimately declares a JSON Schema property literally named `cache_control` is not corrupted (`stripKey`'s `inProperties` tracking, `anthropic.go:505-547`). (`internal/router.TransformerOptions`, `internal/router/router.go:108-113`, maps the config flag the same way for any caller that goes through `internal/router` directly.)

**Q6. Why must `api_base_url` be the complete URL rather than just a host?**

Because the proxy client posts to it **verbatim** — it never appends a path. `config.Provider`'s doc comment spells this out explicitly: the field is documented as already being the complete chat-completions endpoint, e.g. `https://api.deepseek.com/chat/completions` (`internal/config/config.go:48-51`), and `proxy.Client.Do` builds the outgoing request straight from it with no suffixing (`internal/proxy/proxy.go:49-53`, `66`). Configuring just the host would send requests to the provider's root URL and fail identically for every request.

**Q7. What happens if `config.json` is missing entirely?**

Nothing fails. `Load()` treats a missing file as an empty, valid config — zero providers, empty routes — and returns no error, so the gateway can still boot and report "nothing configured" through `/health`/`/ready` rather than refusing to start outright (`internal/config/config.go:170-176`, tested at `internal/config/config_test.go:70-78`).

**Q8. What happens if `config.json` exists but is malformed JSON?**

That **is** a hard error. A missing file is a legitimate "not configured yet" state; a broken file is not — silently continuing with a partially-parsed config risks routing real requests to the wrong upstream with the wrong credentials, which is worse than refusing to start (`internal/config/config.go:178-180`, tested at `internal/config/config_test.go:82-86`).

**Q9. What happens if the JSON is valid but semantically wrong** (e.g. a route pointing at a provider that doesn't exist)?

`Load()` calls `Validate()` after parsing, which also errors out — for a duplicate provider name, a missing/non-empty-but-non-http(s) `api_base_url`, a malformed route string, or a route naming a provider absent from `Providers[]` (`internal/config/config.go:182-186`, `190-233`; all six cases are individually tested at `internal/config/config_test.go:88-108`).

**Q10. How does the router decide which provider and model to use for a request?**

`internal/router.Select` implements the policy (`internal/router/router.go:26-63`):

0. If the request's `model` field itself uses **explicit-selector syntax** — `"provider,model"` or `"provider/model"` — that pins the exact upstream the caller asked for and wins over every rule below, including the haiku heuristic (`internal/router/selector.go:39-92`). Claude Code's own requests never use this syntax (its `model` is always one of its own tier ids); this exists for callers that want to bypass server-configured routing for a single request — e.g. a debugging client, or `claude_toolkit`. A malformed selector, or one naming a provider/model that doesn't exist, fails loudly rather than silently falling through to `Router.default`.
1. Otherwise, if the model id contains `"haiku"` (case-insensitive — Claude Code's cheap/background tier uses ids like `claude-3-5-haiku-20241022`, which *contain* rather than *equal* the tier name) **and** `Router.background` is set, use that route.
2. Otherwise use `Router.default`.
3. If the resulting route string is empty — no `Router` block at all — fall back to the **first** provider in `Providers[]` and the **first** entry in its `Models` list.
4. Any failure along the way (unknown provider referenced, malformed route string, empty providers list, or a fallback provider with no models) returns an explicit error rather than guessing.

Every gateway launched via `cmd/ccr` (`start`/`ui`/`serve`/`web`) uses exactly this — `cmdServe` calls `gw.WireDefaults(0)` before `Start()` (`cmd/ccr/serve.go:44-51`), which installs a small adapter, `routerAdapter`, that delegates straight to `router.Select` (`internal/gateway/wiring.go:29-44`). See Q10a for the one case where this is *not* what runs.

**Q10a. When would a gateway NOT use the full `internal/router.Select` behaviour?**

Only if you build and run `internal/gateway` as a **library**, calling `gateway.New(cfg, opt)` yourself without also calling `srv.WireDefaults(0)` before `Start()`. `internal/gateway/messages.go` deliberately does not import `internal/router`/`internal/proxy` directly — it declares its own narrow `Router`/`Upstream` interfaces with minimal in-package default implementations (`defaultRouter`, `Router.default`-only, no haiku/background awareness; `defaultUpstream`, a plain `net/http` call) so the package compiles and serves correctly on its own (`internal/gateway/messages.go:30-50`, `56-93`). `internal/gateway/wiring.go` is what adapts the real `internal/router`/`internal/proxy` onto those seams, via `Server.WireDefaults` (`internal/gateway/wiring.go:61-71`) — and `cmd/ccr` always calls it. Skip that call in your own code and you silently get the minimal built-ins instead.

**Q11. What are `Router.think` and `Router.longContext` for?**

They exist in the config schema and are validated exactly like `default`/`background` (each must parse as `"provider,model"` and reference a real provider — `internal/config/config.go:217-231`), matching the upstream Node config's fields. Their routing status differs, though:

- **`longContext` is live.** `router.Select` estimates a request's prompt token footprint — it sums the byte lengths of the messages, system block(s), and each tool's name/description/input schema and divides by ~4 chars/token, deliberately excluding `max_tokens` (an output budget, not input) — and when that estimate exceeds `DefaultLongContextThreshold` (60000) **and** `Router.longContext` is set, it routes there (`estimateTokenCount`, `internal/router/selector.go:112-138`; `chooseRoute`, `internal/router/router.go:164-175`). This is checked **first** in the precedence, ahead of `background`/`think`, because an oversized prompt may not physically fit the smaller-tier models. As of v0.4.5 this applies to the OpenAI-inbound facade too: `POST /v1/chat/completions` estimates its own request size via `routingRequestFromOpenAI` (measuring each message's text content plus tools; `image_url` data-URIs are excluded, since a base64 image's byte length is unrelated to its token cost), so `longContext` triggers there symmetrically with `/v1/messages` when configured.
- **`think` is live (v0.4.0).** The routing branch is real and unit-tested (`chooseRoute` returns `Router.Think` when the thinking signal is set and the route is configured), and `translate.AnthropicRequest` now models Anthropic's `thinking` request field (`internal/translate/anthropic.go:51-60`), so an incoming `thinking` block survives decoding: `requestWantsThinking` returns true when the field is present and carries a non-null value, and a `POST /v1/messages` request that asks for extended reasoning routes to `Router.Think` when it is configured (`internal/router/selector.go:150-174`). An omitted field, or an explicit `"thinking": null`, means the client did **not** ask for thinking, so routing is unaffected. No proxy signal (e.g. `max_tokens`) is substituted, because inferring "thinking" from an unrelated field would misroute ordinary requests. **Scope caveat:** unlike `longContext` (which now fires for both inbound paths — see above), `think` fires only for the Anthropic-inbound path; the OpenAI-inbound facade (`POST /v1/chat/completions`) has no `thinking` field to derive the signal from, so such a request is never routed to `Router.think`.

The full documented precedence (first match wins): an explicit `"provider,model"` selector > `longContext` > `background` (haiku tier) > `think` > `default` > an unambiguous bare model (only when no route is configured) > the first-provider fallback (`internal/router/router.go:20-107`).

**Q12. How are image (vision) content blocks translated, and what happens if one can't be?**

Anthropic `type: "image"` content blocks (base64 or URL `source`) are converted to OpenAI `image_url` content parts by `convertImageBlock` (`internal/translate/anthropic.go:237-280`, tested at `internal/translate/vision_test.go`):

- A `base64` source must name one of the four raster media types Anthropic's API documents — `image/png`, `image/jpeg`, `image/gif`, `image/webp` — and decode successfully; the decoded size is checked **before** decoding against a 20MB cap (`base64.StdEncoding.DecodedLen`, so an oversized payload never causes an unbounded allocation).
- A `url` source is passed through as the `image_url.url` directly.
- Anything else — a missing source, an unrecognised `source.type`, a disallowed media type, an over-cap payload, or data that doesn't actually decode as base64 — is a named, explicit `400` error, never a silent drop. The reasoning for erroring rather than dropping predates vision support and still applies: letting the model answer confidently about a picture it never actually saw is a worse failure mode than a loud, immediate error.
- Because an image cannot be represented inside a plain string at all, a message that contains at least one image is sent upstream with **array-shaped** `content` (`OpenAIContentPart[]`); a text-only message still uses the plain-string form some upstreams require (`internal/translate/anthropic.go:420-427`).
- Images inside `tool_result` content (e.g. a computer-use tool returning a screenshot) are handled the same way, via `convertToolResultContent` (`internal/translate/anthropic.go:291-335`).

Vision passthrough is implemented, in other words — this used to be a hard "not supported yet" error for every image, and is not any more.

**Q13. How does the system prompt survive translation?**

Anthropic carries it as a top-level `"system"` field (string or block array); OpenAI has no equivalent field — it expects a `role: "system"` message at the head of the `messages` array. `AnthropicToOpenAI` flattens the (possibly block-array) system field to plain text and, if non-empty, prepends it as the first OpenAI message (`internal/translate/anthropic.go:355-365`). Losing this conversion would silently strip the model's entire instruction set — the code comment calls this out as the single highest-stakes conversion in the file (`internal/translate/anthropic.go:8-13`).

**Q14. How are Anthropic tool calls translated?**

Anthropic represents them as `tool_use` content blocks inside an assistant message and `tool_result` blocks inside a user message. OpenAI instead uses `message.tool_calls` (with JSON arguments encoded as a **string**, not an object — an OpenAI quirk that must be preserved exactly or upstreams reject the payload) plus separate `role: "tool"` messages keyed by `tool_call_id`. `tool_result` blocks are emitted as their own message **before** the remaining content of that turn, matching OpenAI's expected ordering (`internal/translate/anthropic.go:17-19`, `392-408`, `291-335` for `convertToolResultContent`, tested at `internal/translate/anthropic_test.go:87-140`).

**Q15. What is `EnsureToolParameters` for?**

Some upstreams (Poe, per the code comment) reject a tool definition that has no `parameters`/`input_schema` object with a misleading `"Field required"` error rather than treating it as "no parameters." When `Options.EnsureToolParameters` is set, a tool with an empty schema gets `{"type":"object","properties":{}}` backfilled; it is opt-in, so a caller that doesn't need it gets the field left absent exactly as sent (`internal/translate/anthropic.go:157-160`, `462-464`, tested at `internal/translate/anthropic_test.go:144-171`). It is not currently wired to a named config transformer the way `cleancache`/`streamoptions` are (`internal/router/router.go:108-113` only maps those two) — how it gets enabled per-provider is **PLANNED**.

**Q16. Does the gateway remove `Content-Length` on compressed responses — and why?**

Yes. Once a response body is brotli- or gzip-encoded, its length differs from the original, so any `Content-Length` computed beforehand is now wrong; the middleware explicitly deletes the header on a compressed response, because a stale value would make clients truncate the body or hang waiting for more bytes that never arrive (`internal/gateway/compress.go:103-107`, tested at `internal/gateway/gateway_test.go:124-133`).

**Q17. Why does `Alt-Svc` only appear when HTTP/3 is enabled?**

Because advertising `Alt-Svc: h3=...` promises clients they can upgrade to an HTTP/3 endpoint that, if HTTP/3 isn't actually being served, doesn't exist — a broken promise. The middleware that sets the header is only registered when `Options.EnableHTTP3` is true (`internal/gateway/gateway.go:154-156`), and this is asserted directly by test: no `EnableHTTP3` → no `Alt-Svc` header at all (`internal/gateway/gateway_test.go:176-184`).

**Q18. Why does the upstream proxy client use a response-header timeout instead of an overall request timeout?**

Because Claude Code's requests can legitimately stream for a long time — the model keeps generating, and the body keeps arriving in SSE chunks. `http.Client.Timeout` bounds the *entire* request including reading the body, which would cut a slow-but-healthy stream short once it outlives a fixed budget. The standalone `internal/proxy.New` sets only `Transport.ResponseHeaderTimeout`, which catches a genuinely unresponsive upstream (no headers within the timeout) while never truncating a legitimate stream (`internal/proxy/proxy.go:26-44`). This is explicitly tested: a stream configured to run five times longer than the client's timeout still delivers every chunk, because only the *header* wait is time-bounded (`internal/proxy/proxy_test.go:219-266`).

Separately, `handleMessages` itself *also* wraps the request `context` in a `context.WithTimeout(ctx, UpstreamTimeout)` for **non-streaming** calls only, leaving streaming calls with no context deadline added at that layer at all (`internal/gateway/messages.go:248-256`) — this applies regardless of which `Server.Upstream` is wired in, because it happens before `Upstream.Do` is even called, and it also bounds every retry attempt the retry loop makes for that request (see Q28), not just a single call. So on a CLI-launched gateway (which wires in `internal/proxy.Client` via `WireDefaults` — see Q10a), a non-streaming call is bounded by the *stricter* of the two (in practice, the context deadline, since it covers the whole call including body read), while a streaming call is bounded only by `proxy.Client`'s `ResponseHeaderTimeout` — enough to catch an upstream that never responds at all, without ever truncating a stream that has actually started. A gateway built as a library without `WireDefaults` has no such protection on streaming calls at all, since its built-in `defaultUpstream` sets no timeout of its own.

**Q19. Can an upstream error response leak my provider API key?**

No, by design and by test. `proxy.Client.Do` never wraps the outgoing `*http.Request` (which carries the `Authorization: Bearer <key>` header) into any returned error text — it only ever names the provider and the URL, which contain no secret. This is exercised against connection-refused, malformed-URL, and unresolvable-host failures with a real secret string, asserting the key text never appears in the error (`internal/proxy/proxy.go:61-64`, `internal/proxy/proxy_test.go:175-217`).

**Q20. What's the difference between `GET /health` and `GET /ready`?**

`/health` is a liveness probe: it always returns `200` once the process is serving HTTP, along with a `providers` count — it says nothing about whether any provider is actually usable. `/ready` is a readiness probe: it only returns `200` when the router could actually resolve a request — i.e. at least one provider is configured **and** `Router.default` is non-empty — returning `503` with a specific reason string otherwise (`internal/gateway/gateway.go:160-182`, tested at `internal/gateway/gateway_test.go:135-161`). Deliberately, both are unauthenticated and dependency-free, so an external supervisor can distinguish "process up" from "upstream reachable" without needing credentials.

**Q21. Why is the default bind address `127.0.0.1` and not `0.0.0.0`?**

Because that is what Claude Code and the existing `claude_toolkit` deployments already expect: a local, loopback-only gateway, and — more fundamentally — because the gateway holds live provider API keys and sends them upstream on every request, so listening on every interface has to be a deliberate, explicit act, never the default. `New()` only applies the `127.0.0.1` default when `Options.Host` is left empty (`internal/gateway/gateway.go:12-16`, `91-94`); `cmd/ccr` now exposes this as `--gateway-host`/`CCR_GATEWAY_HOST` (`cmd/ccr/flags.go:38-43`) for operators who need to change it — most commonly inside a container, where `127.0.0.1` is the *container's own* loopback and a published port can never reach a process still bound to it. Exposing it beyond loopback on a bare host (not a container) is the operator's explicit choice, and `docs/ADMIN_MANUAL.md` recommends putting an authenticating reverse proxy in front if you do.

**Q22. Why is the default port 3456, and can it be changed?**

3456 matches the Node implementation and every config already written by existing tooling — changing the *default* would break installed setups for no benefit (`internal/gateway/gateway.go:106-108`, comment at `internal/gateway/gateway_test.go:194-197`). It can, however, be overridden per-launch via `--gateway-port`/`CCR_GATEWAY_PORT` (`cmd/ccr/flags.go:29-37`) — useful when something else on the host already holds 3456 (commonly the Node `ccr` this project reimplements). See the next question.

**Q22a. What are `--gateway-host`/`--gateway-port`, and how are they different from `--host`/`--port`?**

`--host`/`--port` (env `CCR_WEB_HOST`/`CCR_WEB_PORT`) configure `cmd/ccr`'s own **management** control-plane server (default `127.0.0.1:3458` — Q27). `--gateway-host`/`--gateway-port` (env `CCR_GATEWAY_HOST`/`CCR_GATEWAY_PORT`) configure the **gateway** itself (default `127.0.0.1:3456` — the one Claude Code talks to). The two pairs are independent by design — setting one never moves the other (`cmd/ccr/gatewayhost_test.go`'s `TestManagementAndGatewaySettingsAreIndependent` asserts exactly this) — and an explicit flag always wins over its environment variable, the same precedence rule `--host`/`--port` already followed (`cmd/ccr/flags.go:50-129`).

Two things worth knowing before relying on `--gateway-host`/`--gateway-port`:

- They work with `ccr serve`/`web` (which construct the gateway directly, `cmd/ccr/serve.go:54-62`), and **`ccr start`/`ui` now forward them too**: `serveChildArgs`, the function that builds the detached `serve` child's argument list (`cmd/ccr/service.go:196-234`), forwards `--gateway-host`/`--gateway-port` alongside `--host`/`--port`/`--gateway`/`--open`, the TLS/HTTP3 flags, and `--max-attempts` (when set) — previously only the first four survived, which for TLS meant `ccr start --tls-cert …` silently served plaintext. The environment-variable form (`CCR_GATEWAY_HOST`/`CCR_GATEWAY_PORT`) still works too, as it always has, since the detached child inherits the parent's environment regardless.
- The stock `Dockerfile` at the repository root does not pass `--gateway-host 0.0.0.0` in its `CMD` — see `docs/ADMIN_MANUAL.md` §1.2 for what that means for container users today.

**Q23. A route string's model id has a comma in it — does that break parsing?**

No. `SplitRoute` splits on the **first** comma only; everything after it — commas included — is the model id (`internal/config/config.go:239-249`). This is explicitly tested with `"prov,vendor/model,v2"` correctly parsing to provider `prov`, model `vendor/model,v2` (`internal/config/config_test.go:110-124`).

**Q24. Does this project contain any code copied from the Node original?**

No. It is explicitly documented as a **clean-room** reimplementation: the wire formats, CLI grammar, config layout, and default ports are reproduced *deliberately* (for compatibility with existing installs), but no upstream source was copied (`NOTICE:1-11`). The upstream MIT licence text is retained verbatim in `LICENSE-UPSTREAM-MIT` purely for attribution.

**Q25. Is there a real CLI, and can I `go install` it, or download a release?**

There is a real, tested CLI (`cmd/ccr`: `start`/`ui`/`serve`/`web`/`stop`, plus a `<profile>` grammar — see Q29 and `docs/USER_GUIDE.md` §4). `v0.1.0` is tagged and published as a [GitHub release](https://github.com/vasic-digital/claude-code-router/releases/tag/v0.1.0), with cross-compiled `linux`/`darwin`/`windows` × `amd64`/`arm64` archives and a `checksums.txt` — download an artifact from there, or build from source: `go build -o bin/ccr ./cmd/ccr`. See `docs/RELEASE.md` for the release process and checksum-verification steps.

**Q26. What is `ccr start`/`ui`/`serve`/`stop`, and why does `ccr <anything-else>` print a "Profile not found" error?**

`cmd/ccr` is a clean-room reimplementation of the upstream Node CLI's **v3.0.6** grammar (`cmd/ccr/main.go:1-19`) — `start`/`ui` launch a detached background service (writing a pidfile + log to `~/.claude-code-router/`), `serve` (alias `web`) runs the same thing in the foreground, and `stop` terminates whatever `start`/`ui` launched. Anything else typed as the first argument is treated as a **profile name or id** per the real CLI's grammar (`ccr <profile-name-or-id> [cli|app] [-- <agent args>]`). This reimplementation does not yet carry a profile store, so every such invocation deterministically prints `Profile "<name>" was not found or is disabled.` to stderr and exits `1` — a faithful, tested match for the upstream CLI's own observed behaviour on an unknown profile, not a bug (`cmd/ccr/main.go:88-94`, tested at `cmd/ccr/main_test.go:43-65`). This exact behaviour matters to `claude_toolkit`, which greps `ccr --help` output for the literal substrings `ccr start` and `ccr serve` to confirm it's talking to a compatible router (`cmd/ccr/main.go:1-5`, tested at `cmd/ccr/main_test.go:12-24`).

**Q27. Why are there two separate HTTP servers (ports 3456 and 3458), and can I turn one off?**

Port 3456 is the Anthropic-compatible **gateway** (`internal/gateway.Server`) — the one Claude Code actually talks to, now independently relocatable via `--gateway-host`/`--gateway-port` (see Q22a). Port 3458 is a small **management** control-plane server that `cmd/ccr` runs alongside it (`cmd/ccr/management.go`), described in its own code comment as deliberately minimal: today it's just an own-shaped `/health` and a placeholder HTML page, with a real web UI explicitly out of scope for now, relocatable via `--host`/`--port`. You can skip starting the *gateway* with `--no-gateway`, but the management server always starts — there is no flag to disable it (`cmd/ccr/serve.go:44-70`).

**Q28. Why doesn't this router support explicit per-request provider selection, retry/fallback, a corporate outbound proxy, or protocols other than Anthropic Messages → OpenAI chat-completions?**

It's a deliberately small, single-purpose reimplementation, not a line-for-line port of the much larger upstream Node product — but several of these have since landed, at least partially, so treat each on its own merits rather than as a blanket "not supported":

- **Explicit per-request provider selection**: implemented and **live** on the CLI-launched gateway. If a request's `model` field itself is shaped like `"provider,model"` or `"provider/model"`, `router.Select` resolves it directly, overriding `Router.default`/`Router.background` (`internal/router/selector.go`, wired into `Select` as rule 0 — `internal/router/router.go:26-30`, `50-58`; tested at `internal/router/router_test.go`). See Q10. (The related bare-model case is now handled too: when no `Router.default` is set, a bare model served by exactly one provider resolves to it, a bare model served by two or more is a loud ambiguity error, and a configured `Router.default` always wins — `internal/router/router.go:72-86`, `internal/router/selector.go`'s `resolveBareModel`.)
- **Corporate/authenticated outbound proxy**: **implemented and live**. `HTTP_PROXY`/`HTTPS_PROXY`/`NO_PROXY` from the environment are still honoured automatically by every `proxy.Client`, including the one `WireDefaults` installs (`internal/proxy/upstream_proxy.go:30-62`, `internal/proxy/proxy.go`'s `New`) — that stays the right mechanism for an **unauthenticated** proxy. For an **authenticated** proxy, a top-level `proxy` block in `config.json` (`ProxyConfig` — `url`/`username`/`password`, all three required together, `internal/config/config.go:163-185`) is now read by `Server.WireDefaults`, which builds the client via `proxy.NewWithUpstreamProxy` (`internal/proxy/upstream_proxy.go:104-161`) and routes every upstream request through it with HTTP Basic auth to the proxy, overriding the ambient `HTTP_PROXY`/`HTTPS_PROXY` (`internal/gateway/wiring.go:72-90`). A partial block is rejected at config load and by `ccr config validate` (`ErrProxyIncomplete`, `internal/config/config.go:358-368`); the proxy `password` is redacted to `[REDACTED]` by `ccr config show`, same as a provider `api_key`. See `docs/USER_GUIDE.md` §2.5 and `docs/ADMIN_MANUAL.md` §5.
- **Retry/fallback on a failed upstream call**: **implemented and live**. `internal/router/fallback.go`'s classification/backoff policy (`ClassifyStatus`, `ClassifyTransportError`, `FallbackRetryDelayAfterStatus`/`...AfterNetworkError`, honouring a `Retry-After` header) is now actually driven by a retry loop: `handleMessages` calls `doUpstreamWithRetry` (`internal/gateway/messages.go:319-416`) instead of making a single `s.Upstream.Do` call directly. It retries up to `Options.MaxAttempts` times total (default 3: one initial attempt plus two retries — `internal/gateway/gateway.go:48-54`, `65-68`), backing off between attempts, and **never** retries a `Terminal`-classified failure (e.g. a `401` — retrying it would only burn quota on a request that will fail identically every time). Once the budget is exhausted or a `Terminal` failure occurs, the response is forwarded exactly as before (see Q30). The attempt budget is now CLI/env-configurable: `--max-attempts <n>` / `CCR_MAX_ATTEMPTS` (must be `>= 1`, `cmd/ccr/flags.go`), forwarded to `ccr start`/`ui`'s detached `serve` child too (`cmd/ccr/service.go:196-234`).
- **Anthropic-native upstreams**: now **supported outbound**. A provider can set `"protocol": "anthropic"` (or have it inferred from an `api.anthropic.com`/`*.anthropic.com` host or an `/anthropic` path segment), and its Anthropic-shaped request is passed through **unchanged** (`internal/translate.AnthropicPassthrough`) and the response relayed verbatim, rather than translated to OpenAI shape — wired in `handleMessages` (`internal/gateway/messages.go:233-295`). See Q22a's sibling on the `protocol` field.
- **OpenAI chat-completions inbound**: now **served** too. Besides `POST /v1/messages`, the gateway accepts `POST /v1/chat/completions` (and both under the `/proxy/v1/...` alias) via an OpenAI-compatible facade (`internal/gateway/openai_inbound.go`), dispatched by a path→protocol classifier (`internal/gateway/protocol.go`). It is a near-passthrough to the (OpenAI-shaped) routed provider; an OpenAI-inbound request that routes to an Anthropic-native provider is an explicit `501`, since OpenAI→Anthropic reverse translation is not implemented.
- **OpenAI Responses API and Gemini inbound**: recognised by the classifier but **not yet served** — no routes are registered for them, so they `404` today.

`test/PORTING-MATRIX.md` documents the fuller picture (though some of its GAP rows for the items above predate this landing and are now stale relative to the code — trust the code and the test names above over that file's prose where they disagree). It also lists an entire N/A tier this router never intended to replicate: Electron-style core/gateway process split, billing telemetry, ToolHub/MCP, OAuth provider-plugin auth, a router rules DSL, "Fusion" vendor routing, hosted web search.

**Q29. Does the gateway check any inbound `Authorization`/API-key header from the client?**

Yes, and it is now switchable from the CLI. `internal/gateway.RequireAPIKey(keys []string)` returns a `gin.HandlerFunc` that accepts either `Authorization: Bearer <key>` or `x-api-key: <key>` (Anthropic's own SDKs send the latter), compared with `crypto/subtle.ConstantTimeCompare` so response timing can't leak how much of a guessed key was correct, and rejects everything else with a fixed `401` message that never echoes what the client sent (`internal/gateway/auth.go`). `internal/gateway/gateway.go`'s route table installs it once and applies it to **all four** completion routes — `/v1/messages`, `/proxy/v1/messages`, `/v1/chat/completions`, `/proxy/v1/chat/completions` (`internal/gateway/gateway.go:362-367`), never just `/v1/messages` alone — as **route-scoped** middleware, deliberately not `s.eng.Use(...)`, so `GET /health`/`GET /ready` stay always-open regardless of auth configuration and a supervisor can keep probing liveness/readiness no matter what.

`Options.APIKeys` (`gateway.go:60-68`) is now populated by `cmd/ccr` from a repeatable `--api-key <key>` flag or the comma-separated `CCR_API_KEYS` environment variable (`cmd/ccr/flags.go`) — a `--api-key` flag value **replaces** the `CCR_API_KEYS` list wholesale rather than merging with it. Prefer `CCR_API_KEYS` (or, for `ccr start`/`ui`, its environment form): a `--api-key` flag value is visible in the process list (`ps`), whereas `start`/`ui` forward configured keys to the detached `serve` child exclusively through the inherited `CCR_API_KEYS` environment, never through argv (`cmd/ccr/service.go:107-114`).

But the accepted-key list still **defaults to empty**, and `RequireAPIKey`'s own documented behaviour for an empty key list is "disable authentication entirely, exactly as if this file did not exist" — required for backwards compatibility, since the toolkit driving this gateway today sends no client key at all, on a loopback-only listener, and a default-deny would break every existing caller the moment authentication was wired in.

Net effect: **a CLI-launched gateway is still unauthenticated by default today**, same as before — only now there is an explicit switch (`--api-key`/`CCR_API_KEYS`) an operator can flip. Anyone who can reach the gateway's port with no keys configured can send it requests billed to your configured provider keys. See `docs/ADMIN_MANUAL.md` §5 for mitigations (configure `CCR_API_KEYS`, bind to loopback, or put an authenticating reverse proxy in front).

**Q30. What does a `POST /v1/messages` error response look like, and which HTTP status codes does it use?**

Every error — whether it originates in the gateway itself (bad JSON, no route, a translation failure like an unsupported image source), forwarded from the upstream provider, or produced by the retry loop giving up — uses the same Anthropic-style shape: `{"type":"error","error":{"type":"<error_type>","message":"<message>"}}` (`internal/gateway/messages.go:434-446`). Gateway-originated errors use `400` for a malformed/untranslatable request, `503` when no route can be resolved, `502` for an upstream transport failure that exhausted its retries, `500` only for the (should-be-rare) case of failing to encode the already-validated outgoing request, and (new, from the retry loop) `504` when the request's context ends — client disconnect, or the non-streaming deadline — while an attempt is in flight or waiting to retry (`internal/gateway/messages.go:344-416`). A **non-2xx upstream response**, once the retry budget is exhausted or the status is `Terminal` (not worth retrying), is different: its exact status code is preserved and forwarded as-is — `forwardUpstreamError` maps the status to an Anthropic error `type` (e.g. `401`→`authentication_error`, `403`→`permission_error`, `404`→`not_found_error`, `429`→`rate_limit_error`, `400`/`422`→`invalid_request_error`, any `5xx`→`api_error`) rather than collapsing everything to a generic `502`, because Claude Code's retry/backoff behaviour depends on seeing the real status code (`internal/gateway/messages.go:448-504`, tested at `internal/gateway/messages_test.go:224-259`). Full reference: `docs/API.md`.

**Q31. How does a streamed response map OpenAI SSE chunks to Anthropic SSE events?**

`streamAnthropicSSE` reads the upstream's `data: {...}` lines (terminating on `data: [DONE]`) and re-emits the standard Anthropic Messages event sequence: one `message_start`, then per content block a `content_block_start` / one-or-more `content_block_delta` / `content_block_stop` triplet (text uses `text_delta`, tool-call argument fragments use `input_json_delta`), then a closing `message_delta` (carrying the mapped `stop_reason` and final token usage) and `message_stop` (`internal/gateway/messages.go:621-763`). Every individual event is flushed the moment it's written — not batched — which is asserted directly by test via a flush-counting response recorder (`internal/gateway/messages_test.go:15-27`, `180-185`). A malformed individual chunk line from the upstream is skipped rather than aborting the whole stream, since by the time a bad line arrives the response has already started and there is no status code left to change (`internal/gateway/messages.go:710-714`). Note streaming responses are only ever handed to `streamAnthropicSSE` after the retry loop has already produced its final answer (Q28) — a retry can never corrupt an in-flight SSE conversation, because none of the intermediate failed attempts ever reach this function.
